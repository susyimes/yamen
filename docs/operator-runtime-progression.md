# Operator Runtime Progression

This document records the staged path from contracts to real OpenClaw execution.

## Confirmed architecture

- Yamen repo = 规则层
- OpenClaw = 运行层
- yamen-operator = 内部执行 skill

## Current completed pieces

### Rule layer
- case contract
- transitions contract
- entry output contract
- operator status contract
- prefect report contract

### Operator skill layer
- execution flow reference
- failure handling reference
- summary/report contract reference
- session payload reference
- minimal test reference
- host actions reference
- bridge driver reference

### Local validation layer
- happy-path smoke runner
- failure smoke runner

## Next runtime progression

### Stage A
Keep using local smoke runners to freeze contracts.

### Stage B
Use `runtime/openclaw-bridge-relay.js` as the manual bridge driver.

At this stage:
- request files are generated by provider/runtime
- operator or human inspects request
- OpenClaw session action is executed manually/semi-manually
- role JSON is written back as response file

### Stage C
Add a thin adapter that prints or materializes exact OpenClaw session-tool payloads.

### Stage D
Only then wire fully automatic session-tool dispatch.

## Design preference

Prefer:
- simple stop/report failures
- thin bridge logic
- strict separation of rule layer and runtime layer

Avoid:
- putting orchestration policy into relay scripts
- overbuilding retries before the first real bridge path works
