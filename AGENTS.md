# AGENTS.md - Yamen Agent Infrastructure

Yamen 是一个以中国古代县衙 / 州府制度为灵感的轻量多 Agent 架构。

## 目标

这不是“大而全”的朝廷系统，而是一个适合日常事务和中小任务的少 Agent 协作模型。

核心要求：
- 默认短链路
- 默认少角色
- 默认快处置
- 风险任务才加复核
- 超复杂任务应升级到更重制度（如 Edict）

## 核心角色

- `menfang`：门房 / 受理
- `xianling`：县令 / 主官决断
- `zhubu`：主簿 / 任务单与文书
- `kuaishou`：快手 / 承办执行
- `dianshi`：典史 / 风控复核
- `zhangfang`：账房 / 台账与数据（可选）

## 运行模式

Yamen 的制度层保持少角色，但运行层允许更细的 mode：

- `triage`：受理、分类、接受/驳回、选路
- `plan`：拆任务、补边界、形成执行说明
- `execute`：实际执行
- `debug`：查链路、查状态、查 contract、定位失败点
- `review`：风险把关、放行、退回

推荐映射：
- `entry` 承载 `triage`
- `zhubu` 承载 `plan`
- `kuaishou` 承载 `execute`
- `dianshi` 承载 `debug` / `review`

原则：**mode 是运行能力，不额外膨胀成新官职。**

## 基础流程

### 直办
门房 → 县令 / 快手 → 回禀

### 立案
门房 → 县令 → 主簿 → 快手 → 回禀

### 复核案
门房 → 县令 → 主簿 / 快手 → 典史 → 县令回禀

## 执行席位规则

### 快手 = delegated execution seat

快手是正式承办位，但运行时可以有三种形态：
- 单人执行
- sub-agent 执行
- 并行执行（前提是任务可拆、可合并、可验收）

规则：
- 对制度层仍统一记作 `kuaishou`
- 不把每个执行 worker 都提升成新角色
- 所有 delegated/parallel 输出都要先收拢为一个正式 `kuaishou` 结果

### 典史 = debug / review 主承载位

典史不负责默认主执行，主要负责：
- `debug`：查链路、查状态、查 contract、查 role 冲突、查无法收口原因
- `review`：风险把关、放行、退回修正

## 边界

- 门房不做复杂规划
- 县令不直接承办细节执行
- 主簿不替代快手长期执行
- 快手不自己给自己复核
- 典史不负责主执行，只负责风险和放行
- 出现链路漂移时，应优先进入 `debug`，而不是盲目继续执行

## 输出原则

所有 Agent 输出优先使用结构化字段：
- 标题
- 摘要
- 风险等级
- 处理模式
- 主办角色
- 当前状态
- 阻塞项
- 回禀摘要

## Debug 触发规则

符合以下任一条件时，应触发 `debug` mode：
- 输出不满足 schema / contract
- 连续两步失败
- role 冲突或非法 `next_role`
- 无法通过 `entry` 正常收口
- 用户明确说“这不对，重新定位问题”

## 升级机制

符合以下任一条件时，建议升级到更重制度：
- 跨多个专业域且依赖复杂
- 高风险部署 / 外部发布 / 高权限操作
- 需要正式审议或多轮会签
- 周期较长且需要强审计回放
